# Evo-Golf: The Genetic Swing — MVP PRD

## Project overview

Evo-Golf is a 3D golf game where players don't control the swing directly — they **evolve** it using genetic programming (GP). A population of "swing programs" (represented as expression trees) launches golf balls simultaneously. The player guides evolution across generations — selecting parents, watching crossover and mutation, and converging on a swing that sinks the ball.

The game teaches GP intuitively by making each pillar (Initialization, Selection, Crossover, Replication, Mutation) a visible, interactive game phase.

**Tech stack:** SpacetimeDB (TypeScript module) + React Three Fiber (Vite + TypeScript client). SpacetimeDB runs in Docker for local dev.

One row per GP generation.

| Column | Type | Notes |
|--------|------|-------|

| `final_z` | `f64` | |

High-frequency simulation output. One row per time step per ball.

| `wildcard_genome_id` | `u32` | 0 if none — the player's "sponsored" ball |

## SpacetimeDB Module — Reducers

### GP Genome Tree Format

The genome is a program tree serialized as JSON. When evaluated, it produces a set of launch parameters for the golf ball.

```typescript
// Tree node types
type FuncNode = {
  op: 'add' | 'sub' | 'mul' | 'div_safe' | 'sin' | 'cos' | 'if_gt' | 'max' | 'min';
  children: TreeNode[];
};

type TerminalNode = {
  terminal: 'launch_angle' | 'power' | 'spin_x' | 'spin_z' | 'wind_x' | 'wind_z' | 'const';
  value?: number; // Only for 'const'
};

type TreeNode = FuncNode | TerminalNode;
```

**Evaluation:** The tree is evaluated to produce a `SwingParams` object:
```typescript
type SwingParams = {
  launch_angle: number;   // 10–80 degrees, clamped
  power: number;          // 0–1 normalized, clamped
  spin_x: number;         // -1 to 1, side spin
  spin_z: number;         // -1 to 1, backspin/topspin
};
```

The tree is evaluated by recursively resolving nodes. Terminal `launch_angle` returns a base value (45°), `power` returns 0.7, etc. These are then combined by the operators in the tree to produce final params. The GP evolves the tree structure to find optimal combinations.

### Reducer specifications

#### `create_game()`

Creates the initial `golf_course` row for the MVP hole. Called once on first load. Idempotent — no-ops if a course already exists.

Course definition for MVP:
- Tee at (0, 0), hole at (0, 137) — 150 yards straight
- Wind: (0.5, 0.2) — light crosswind
- Par: 3

#### `init_population(hole_id: u32, pop_size: u32)`

**GP Pillar: INITIALIZATION**

Creates a new `generation` row with `phase: 'init'` and `gen_number` incremented from the previous generation (or 0 if first).

Generates `pop_size` random genome trees using **ramped half-and-half**:
1. For depths 2 through `max_init_depth` (default 4), allocate an equal share of the population.
2. Within each depth cohort, half use the **full** method (all branches extend to max depth) and half use the **grow** method (branches terminate randomly between 1 and max depth).
3. Each tree node is randomly chosen: function nodes from `{add, sub, mul, div_safe, sin, cos, if_gt}`, terminal nodes from `{launch_angle, power, spin_x, spin_z, wind_x, wind_z, const}`. Constants are random floats in [-2, 2].

For each genome, also inserts a `golf_ball` row with `state: 'waiting'`.

Logs a `gp_event` with `event_type: 'init'`.

Sets `generation.phase = 'init'`.

**MVP default: `pop_size = 12`** (balances visual clarity with evolutionary diversity).

#### `simulate_shots(gen_id: u32)`

**Core physics simulation.** Evaluates each genome's program tree to produce swing params, then runs a simplified ballistic simulation.

Physics model (simplified for MVP):
1. Evaluate genome tree → get `{launch_angle, power, spin_x, spin_z}`, clamped to valid ranges.
2. Convert to initial velocity vector:
   - `speed = power * MAX_SPEED` (MAX_SPEED = 70 m/s ≈ 156 mph)
   - `vx = speed * sin(launch_angle) * spin_x_factor`
   - `vy = speed * sin(launch_angle)`
   - `vz = speed * cos(launch_angle)`
3. Step simulation at `dt = 0.05s` for up to 200 steps:
   - Apply gravity: `vy -= 9.8 * dt`
   - Apply wind: `vx += wind_x * dt * 0.3`, `vz += wind_z * dt * 0.3`
   - Apply drag: `v *= 0.998`
   - Update position: `pos += v * dt`
   - If `y <= 0`: ball has landed, apply bounce/roll damping, then a few roll steps.
   - Write a `trajectory_point` row for each step.
4. Set `golf_ball.state = 'stopped'`, record `final_x/z`, compute `distance_to_hole`.
5. Compute fitness: `fitness = 1.0 / (distance_to_hole + 1.0)`. Closer = higher fitness (max approaches 1.0).
6. Update `genome.fitness` and `generation.best_fitness / avg_fitness`.
7. Set `generation.phase = 'evaluated'`.

Log a `gp_event` with `event_type: 'simulate'`.

#### `run_selection(gen_id: u32, tournament_size: u32)`

**GP Pillar: SELECTION**

Implements **tournament selection**:
1. Read all genomes for `gen_id`, sorted by fitness descending.
2. Select `num_parents` parents (default: `pop_size / 2`, i.e. 6):
   - Randomly sample `tournament_size` (default: 3) genomes.
   - The one with highest fitness wins the tournament.
   - Mark it as `is_selected = true`.
3. If a `player.wildcard_genome_id` is set and that genome is in this generation, force-select it as an additional parent (human-guided selection).
4. Set `generation.phase = 'selecting'`.
5. Log `gp_event` entries for each selection.

#### `replicate_elite(gen_id: u32)`

**GP Pillar: REPLICATION**

1. Find the genome with the highest fitness in `gen_id`.
2. Create a new `generation` row (next gen) if not already created.
3. Insert an exact copy of the elite genome into the new generation with `origin: 'replication'`, `is_elite: true`, and the same `tree_json`.
4. Log `gp_event` with `event_type: 'replicate'`.

This is elitism — the champion survives unchanged to the next generation.

#### `perform_crossover(gen_id: u32, num_offspring: u32)`

**GP Pillar: CROSSOVER**

1. Read all selected parents (`is_selected = true`) for `gen_id`.
2. For each offspring (default: `pop_size - 2`, reserving 1 slot for elite + 1 for potential mutation-only):
   - Pick two distinct parents randomly from the selected pool.
   - Parse both `tree_json` into tree objects.
   - Select a random subtree position in Parent A's tree.
   - Select a random subtree position in Parent B's tree.
   - Swap the subtrees: replace the chosen subtree in Parent A with the one from Parent B.
   - Serialize the resulting child tree.
   - Insert as a new genome with `origin: 'crossover'`, `parent_a_id` and `parent_b_id` set.
3. Log `gp_event` for each crossover, including parent and child IDs.
4. Set `generation.phase = 'breeding'`.

#### `perform_mutation(gen_id: u32, mutation_rate: f64)`

**GP Pillar: MUTATION**

1. Read all offspring genomes in the NEW generation (those with `origin: 'crossover'`).
2. For each offspring, with probability `mutation_rate` (default: 0.3):
   - Parse `tree_json`.
   - Choose a random mutation type (equal probability):
     - **Subtree mutation**: Replace a random subtree with a newly generated random subtree (depth 1–3).
     - **Point mutation**: Replace a single terminal's value (e.g. change a constant, swap `launch_angle` for `power`).
     - **Hoist mutation**: Replace a subtree with one of its own children (guaranteed to shrink the tree).
   - Serialize the mutated tree, update `tree_json`, recalculate `tree_depth` and `node_count`.
   - Set `origin: 'mutation'`.
3. Log `gp_event` for each mutation.

#### `advance_generation(gen_id: u32)`

Orchestrator reducer. Calls the pipeline in order:
1. `run_selection(gen_id, 3)`
2. `replicate_elite(gen_id)` → creates new generation
3. `perform_crossover(gen_id, pop_size - 1)`
4. `perform_mutation(new_gen_id, 0.3)`
5. Clear old `trajectory_point` rows for previous gen (keep last 2 gens for comparison view).
6. `simulate_shots(new_gen_id)`
7. Set new `generation.phase = 'evaluated'`.

In **manual mode**, the player calls each step individually via the UI. In **auto-evolve mode**, `advance_generation` runs the full pipeline.

#### `set_wildcard(genome_id: u32)`

Updates the calling player's `wildcard_genome_id`. This genome gets a bonus in selection — it's always included as a parent, representing human-guided evolution.

### Connection setup

```typescript
// src/hooks/useSpacetimeDB.ts
import { DbConnection, tables } from '../module_bindings';

export function useSpacetimeDB() {
  const [conn, setConn] = useState<DbConnection | null>(null);

  useEffect(() => {
    const connection = DbConnection.builder()
      .withUri('ws://localhost:3000')
      .withDatabaseName('evogolf')
      .onConnect((ctx) => {
        ctx.subscriptionBuilder()
          .onApplied(() => { /* ready */ })
          .subscribe([
            tables.golfCourse,
            tables.generation,
            tables.genome,
            tables.golfBall,
            tables.trajectoryPoint,
            tables.gpEvent,
            tables.player,
          ]);
      })
      .build();

    setConn(connection);
    return () => connection.disconnect();
  }, []);

  return conn;
}
```

### Component specifications

#### `<CourseGround />`

Renders the golf course as a flat green plane with:
- A tee box marker (small brown rectangle at tee position)
- A hole/pin marker (flag mesh at hole position)
- A yard-line grid (dashed lines every 25 yards)
- A wind direction indicator (arrow overlay)

Reads from `golf_course` table.

MVP: flat `<Plane>` geometry, no terrain displacement. Green material with grid lines via a custom shader or drei's `<Grid>`.

#### `<BallSwarm />`

Renders all `golf_ball` rows for the current generation. Uses `<instancedMesh>` for performance (12 balls).

Color coding by `genome.origin`:
- White: `random` (initial population)
- Gold: `replication` (elite clone)
- Green: `crossover` (offspring)
- Purple: `mutation` (mutated offspring)

When `golf_ball.state === 'waiting'`, balls sit on the tee. When trajectory points exist, `useBallAnimation` interpolates position along the trajectory path over ~4 seconds using `useFrame`.

Clicking a ball sets it as the "selected" ball, highlighting it and displaying its genome in `<GenomeTreePanel>`.

#### `<TrajectoryLines />`

For each `golf_ball` in the current generation, queries `trajectory_point` rows ordered by `step`. Renders as `<Line>` components (from drei) with:
- Color matching the ball's origin color
- Opacity fading from 1.0 (recent) to 0.2 (old)
- The selected ball's trajectory is fully opaque and thicker

#### `<GenomeTreePanel />`

An HTML overlay (via drei `<Html>`) displaying the selected genome's program tree as a visual node graph.

Tree rendering (2D, React components overlaid on the 3D scene):
- Each node is a rounded box: function nodes show the operator name, terminal nodes show the value
- Edges connect parent to children
- Color highlights:
  - Amber glow on subtrees that were involved in crossover (compare `parent_a_id`/`parent_b_id` trees)
  - Purple pulse on nodes that were mutated
  - Gold border on the elite genome

Shows key stats: `tree_depth`, `node_count`, `fitness`, `origin`.

MVP implementation: a simple recursive React component that renders the JSON tree as nested `<div>` elements with flexbox layout. No need for a graph library.

#### `<GPControlPanel />`

Docked at the bottom of the screen. Contains:

1. **Phase indicator**: Shows current `generation.phase` as a step progress bar: Init → Simulate → Evaluate → Select → Breed → Complete
2. **Action buttons** (enabled/disabled based on current phase):
   - "Initialize Population" → calls `init_population` reducer
   - "Launch Balls" → calls `simulate_shots` reducer
   - "Run Selection" → calls `run_selection` reducer
   - "Breed Next Gen" → calls `replicate_elite` + `perform_crossover` + `perform_mutation`
   - "Next Generation (Auto)" → calls `advance_generation`
3. **Generation counter**: "Generation 4 / Best: 0.87 / Avg: 0.42"
4. **Wildcard button**: "Sponsor This Ball ⭐" — calls `set_wildcard` with the selected ball's genome ID
5. **Speed slider**: Controls animation playback speed (client-side only)

#### `<FitnessChart />`

A `recharts` `<LineChart>` showing:
- X axis: generation number
- Y axis: fitness (0–1)
- Two lines: best fitness (solid) and average fitness (dashed)
- Updates in real time as new `generation` rows arrive

#### `<HUD />`

Top-of-screen overlay showing:
- Current hole info (par, distance, wind)
- Generation number
- Population size
- Best ball distance to hole (in yards)
- "Hole in one!" celebration when `distance_to_hole < 0.5`

## Game loop (user flow)

### First-time flow

1. Client connects → calls `create_game()` → course appears
2. Player clicks **"Initialize Population"** → 12 random balls appear on the tee, genome trees visible in side panel
3. Player clicks **"Launch Balls"** → all 12 balls fly simultaneously along their computed trajectories. Some fly wildly off course, a few land near the fairway.
4. Balls stop → fitness scores appear above each ball. The closest ball glows.
5. Player clicks a ball they like → clicks **"Sponsor This Ball ⭐"** (wildcard selection)
6. Player clicks **"Run Selection"** → tournament brackets animate, top performers are highlighted
7. Player clicks **"Breed Next Gen"** → crossover/mutation happens, new balls appear. The genome tree panel shows how parent trees were spliced.
8. Repeat from step 3.

### Auto-evolve flow

Player toggles "Auto-Evolve" → `advance_generation` runs repeatedly with a 2-second visual delay between generations. The player watches fitness climb on the chart and balls converge toward the hole across generations.

### Win condition (MVP)

When any ball achieves `distance_to_hole < 0.5` yards (~18 inches), display a celebration overlay: "HOLE IN ONE! Evolved in N generations." Show the winning genome tree.

## MVP Scope & Constraints

### In scope

- Single flat hole, no terrain, no hazards, basic wind
- 12-ball population per generation
- Full GP pipeline: init, simulate, select, crossover, mutate, replicate
- Manual step-by-step mode AND auto-evolve mode
- Genome tree visualization (2D overlay)
- Fitness chart over generations
- Ball color coding by origin type
- Trajectory line rendering
- Wildcard/sponsor selection (human-guided evolution)
- Single player (but architecture supports multiplayer via SpacetimeDB subscriptions)

### Out of scope (future)

- Multiple holes with progressive difficulty
- Terrain elevation / heightmap displacement
- Water hazards, bunkers, doglegs
- Multiplayer voting / competitive mode
- 3D genome tree visualization
- Sound effects and music
- Mobile touch controls
- Leaderboard / hall of fame
- Advanced GP features: automatically defined functions (ADFs), strongly typed GP, co-evolution

### Performance targets

- 12 balls rendering at 60fps
- ~2,400 trajectory points per generation (12 balls × 200 steps) — well within SpacetimeDB throughput
- Genome tree JSON serialization: < 2KB per genome typical
- Full generation cycle: < 500ms server-side

## Key technical decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Module language | TypeScript | Fastest iteration for GP tree manipulation. First-class SpacetimeDB 2.0 support. |
| Physics simulation | Server-side (reducer) | Deterministic, authoritative. All clients see identical trajectories. |
| Ball animation | Client-side interpolation | Server writes trajectory points; client animates between them using `useFrame`. Decouples sim speed from render speed. |
| Genome storage | JSON string column | SpacetimeDB doesn't support recursive types in table schemas. JSON is flexible and < 2KB per tree. |
| Tree depth cap | Max 6 levels | Prevents bloat. Hoist mutation provides natural tree-size pressure. |
| Docker for SpacetimeDB | Yes | Reproducible local dev. No system-level installation. Volume mount for persistence. |

## Reference: GP pillar demonstration checklist

Each pillar must be **clearly visible and interactive** in the game. This checklist is the acceptance criteria.

- [ ] **Initialization**: Player sees 12 diverse balls spawn. Genome panel shows trees of varying depth/shape. Event log says "Population initialized with ramped half-and-half method."
- [ ] **Selection**: Player sees a tournament bracket or highlight animation. Top performers glow. Wildcard ball gets a star. Event log shows "Tournament selection: genome #3 wins over #7, #12."
- [ ] **Crossover**: Player sees two parent trees side-by-side with a subtree highlighted in each. The subtrees swap. A child tree appears. Event log says "Crossover: genome #3 × genome #5 → genome #15."
- [ ] **Replication**: Player sees the best ball turn gold and duplicate. Its tree is copied verbatim. Event log says "Elite replication: genome #3 (fitness 0.87) copied unchanged."
- [ ] **Mutation**: Player sees some offspring trees flash purple. A subtree shakes and is replaced. Event log says "Subtree mutation on genome #15: replaced node at depth 2."
